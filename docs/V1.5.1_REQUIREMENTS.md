# v1.5.1 API Burst Test 요구사항 명세서

> **버전**: v1.5.1  
> **작성일**: 2024-12-19  
> **참고 모델**: [rakyll/hey](https://github.com/rakyll/hey) - HTTP load generator  
> **제약**: Frontend-only (브라우저 로컬 실행)

---

## 1. 개요

### 1.1 도구 설명

**API Burst Test**는 개발자가 특정 API endpoint에 짧고 강한 Burst 부하를 주어 성능 지표(지연시간 분포, RPS, 에러율, 상태코드 분포)를 즉시 확인할 수 있는 브라우저 기반 부하 테스트 도구입니다.

### 1.2 핵심 가치

- **즉시 사용**: 설치 없이 브라우저에서 바로 부하 테스트 실행
- **프라이버시**: 모든 요청은 사용자 브라우저에서 직접 발생, 서버 경유 없음
- **공유 가능**: URL 링크로 테스트 설정 공유

### 1.3 사용자 스토리

1. 개발자로서 특정 API endpoint에 대해 **짧고 강한 Burst 부하**(N회 / T초 / QPS 제한)를 주고 **지연시간 분포(p50/p95/p99)**, **RPS**, **에러율**, **상태코드 분포**를 즉시 확인하고 싶다.
2. QA/리그레션 상황에서 **동일한 설정을 URL로 공유**(팀원에게 재현)하고 싶다.
3. CORS 제약 때문에 브라우저에서 막히는 케이스는 **Companion Extension이 있을 때만** 우회해서 테스트하고 싶다.

## 2. 악용 방지 및 제약 사항 ⚠️

> **중요**: 브라우저 기반 부하 테스트 도구는 악용될 소지가 있으므로 아래 제약을 **반드시** 적용합니다.

### 2.1 하드 리밋 (Hard Limits)

| 항목 | 최대값 | 근거 |
|------|--------|------|
| **동시 요청 수 (Concurrency)** | 50 | 브라우저 동시 연결 제한 + DDoS 방지 |
| **총 요청 수 (Total Requests)** | 10,000 | 메모리 사용량 제한 + 악용 방지 |
| **테스트 지속 시간 (Duration)** | 60초 | 장시간 부하 방지 |
| **QPS 상한** | 1,000 req/s | 과도한 부하 방지 |
| **요청 타임아웃** | 최소 100ms ~ 최대 60초 | 합리적 범위 제한 |
| **Body 크기** | 1MB | 메모리 보호 |
| **Header 개수** | 20개 | 과도한 헤더 방지 |
| **per-request 샘플 저장** | 최대 1,000개 | 메모리 폭발 방지 |

### 2.2 소프트 리밋 및 경고

| 조건 | 경고 메시지 | UI 처리 |
|------|-------------|---------|
| **Concurrency > 20** | "High concurrency may affect browser performance and target server" | Warning 배지 표시 |
| **Total Requests > 1,000** | "Large number of requests. Ensure you have permission to test this endpoint" | Confirmation 다이얼로그 |
| **Duration > 30초** | "Long-duration test. Monitor browser memory usage" | Warning 배지 표시 |
| **QPS > 100** | "High QPS may be throttled by browser or network" | Info 툴팁 |
| **외부 도메인 대상** | "Testing external domains may violate their terms of service" | 최초 실행 시 경고 다이얼로그 |

### 2.3 필수 경고 문구 (UI 상시 표시)

도구 상단에 다음 경고를 **항상** 표시:

> ⚠️ **Responsible Use Notice**  
> This tool is for testing APIs you own or have explicit permission to test.  
> Unauthorized load testing may violate terms of service or laws.  
> All requests originate from your browser - your IP will be visible to the target server.

### 2.4 실행 전 확인 (Required Acknowledgment)

- 첫 번째 테스트 실행 시 **반드시** 아래 체크박스 확인 필요:
  - [ ] I have permission to test this endpoint
  - [ ] I understand requests will be sent from my IP address
- 이 확인은 **세션당 1회** (브라우저 탭 닫으면 리셋)
- localStorage에 "acknowledged" 플래그 저장 금지 (매번 확인 유도)

### 2.5 Rate Limiting 동작

- 브라우저 탭당 **동시에 1개의 테스트만** 실행 가능
- 테스트 완료 후 **쿨다운 없음** (연속 실행 허용)
- Stop 후 즉시 재시작 가능

## 3. 기능 범위

### 3.1 MVP 범위 (v1.5.1)

#### Target 설정
- URL 입력 (단일 endpoint)
- HTTP Method: GET, POST, PUT, DELETE, HEAD, OPTIONS
- Headers: key/value 다중 입력, 개별 활성화/비활성화 토글
- Body: Raw 텍스트(기본), JSON(헬퍼), Form URL-encoded(헬퍼)
- Basic Auth (username:password)

#### Load Shape
- **Mode A**: Total Requests(N) + Concurrency(C)
- **Mode B**: Duration(Z) + Concurrency(C)
- **Rate Limit** (선택): Global QPS 또는 Worker당 QPS

#### Timeout
- 요청 타임아웃(초), AbortController로 강제 취소

#### Metrics 수집
- **Throughput**: RPS(성공/전체), 총 요청 수, 총 소요시간
- **Latency**: avg/min/max + p50/p90/p95/p99
- **Status Code**: 카운트(2xx/3xx/4xx/5xx, 개별 코드)
- **Error Breakdown**: timeout, network/CORS, aborted, 4xx/5xx, 기타

#### 출력/내보내기
- Summary View (인간 친화적 UI)
- Export JSON
- Export CSV (요약 + 선택적 per-request 샘플링)

#### 세션 관리
- Run / Stop / Reset
- 마지막 설정 localStorage 저장
- 공유 링크(URL fragment) 생성

### 3.2 비범위 (v1.5.1에서 제외)

- 다중 시나리오/스텝(워크플로우 기반)
- Ramp-up/Ramp-down 부하 패턴
- 분산 부하 테스트
- WebSocket/gRPC/native TCP 레벨 부하
- 브라우저 통제 불가 네트워크 스위치:
  - Keep-alive disable/enable
  - 압축 disable
  - 프록시 지정
  - HTTP/2 강제 (브라우저 자동 협상)
- CORS 우회 (Extension 없이는 CORS 허용된 API만 지원)

## 4. UI/UX 디자인 가이드

### 4.1 레이아웃 구조

#### 데스크탑 (lg 이상)
- **2-Column Layout**: 좌측 Config (40%), 우측 Results (60%)
- Config는 스크롤 가능한 패널
- Results는 탭 기반 (Summary / Charts / Details / Export)

#### 모바일/태블릿 (lg 미만)
- **Single Column Layout**: Config → Run Control → Results 순서
- 접이식(Collapsible) 섹션으로 화면 절약
- Config 섹션은 기본 접힘, 탭하여 펼침
- Results는 탭 기반 유지

### 4.2 컴포넌트별 디자인 명세

#### 4.2.1 경고 배너 (Warning Banner)
- 위치: 도구 최상단, 항상 표시
- 스타일: `bg-amber-50 dark:bg-amber-900/20 border-l-4 border-amber-500`
- 아이콘: `AlertTriangle` (Lucide)
- 텍스트: 14px, `text-amber-800 dark:text-amber-200`
- 닫기 버튼 없음 (항상 표시)

#### 4.2.2 URL 입력 필드
- 전체 너비, `h-12` 높이
- Method 선택 드롭다운을 prefix로 통합 (InputGroup 패턴)
- Method 버튼: `w-24`, 배경색으로 메서드 구분
  - GET: `bg-emerald-100 dark:bg-emerald-900/30`
  - POST: `bg-blue-100 dark:bg-blue-900/30`
  - PUT: `bg-amber-100 dark:bg-amber-900/30`
  - DELETE: `bg-red-100 dark:bg-red-900/30`
  - HEAD/OPTIONS: `bg-gray-100 dark:bg-gray-800`
- URL 입력: `font-mono`, placeholder "https://api.example.com/endpoint"

#### 4.2.3 Load Mode 토글
- 2개 버튼 그룹 (SegmentedControl 패턴)
- 선택됨: `bg-blue-600 text-white`
- 미선택: `bg-gray-100 dark:bg-gray-800 text-gray-600 dark:text-gray-400`
- 각 모드 선택 시 관련 입력 필드 표시/숨김

#### 4.2.4 Concurrency/QPS 슬라이더
- 슬라이더 + 숫자 입력 조합
- 슬라이더: `accent-blue-600`, 트랙 `bg-gray-200 dark:bg-gray-700`
- 숫자 입력: 우측에 `w-20` 크기
- 경고 임계값 초과 시 슬라이더 트랙 색상 변경: `bg-amber-200`

#### 4.2.5 Headers 편집기
- 테이블 형태, 각 행: Checkbox (활성화) + Key 입력 + Value 입력 + 삭제 버튼
- "Add Header" 버튼: `+ Add Header`, 테이블 하단
- Key/Value 입력: `font-mono text-sm`
- 빈 행 자동 추가 (마지막 행 입력 시)

#### 4.2.6 Body 편집기
- 탭: Raw / JSON / Form
- CodeMirror 에디터 사용 (JSON 모드 시 syntax highlighting)
- 높이: 데스크탑 `h-40`, 모바일 `h-32`
- GET/HEAD 선택 시:
  - 비활성화 + 오버레이 메시지: "Body is not available for GET/HEAD requests"

#### 4.2.7 Run Control Bar
- **Sticky**: 스크롤 시 상단 고정
- 배경: `bg-white/80 dark:bg-gray-900/80 backdrop-blur-sm`
- 버튼 배치:
  - Run: `bg-blue-600 hover:bg-blue-700 text-white px-6 py-2`
  - Stop: `bg-red-600 hover:bg-red-700 text-white px-6 py-2` (실행 중만 표시)
  - Reset: `text-gray-600 dark:text-gray-400 hover:text-gray-900` (ghost 스타일)
- 진행률 표시 (실행 중):
  - Progress bar: `h-1` 아래에 위치
  - 텍스트: "Running... 1,234 / 10,000 requests" 또는 "Running... 15.2s / 60s"

#### 4.2.8 Results Summary Cards
- 그리드 레이아웃: 데스크탑 4열, 태블릿 2열, 모바일 1열
- 각 카드: `rounded-lg border p-4`
- 주요 지표:
  - **Total Requests**: 큰 숫자 + 작은 라벨
  - **RPS**: 숫자 + `/sec` 서픽스
  - **p95 Latency**: 숫자 + `ms` 서픽스
  - **Error Rate**: 퍼센트, 0%면 녹색, >0%면 적색

#### 4.2.9 Latency Distribution Chart (hey 스타일)
- **Histogram Bar Chart**: 가로 막대
- X축: Latency 범위 (0-0.5s, 0.5-1s, ...)
- Y축: 요청 수
- 막대: `bg-blue-500`, 최대값 막대 기준 상대 너비
- 각 막대 옆에 카운트 숫자 + ASCII art 스타일 `■` 표시 (hey 감성)

#### 4.2.10 Status Code Distribution
- **Horizontal Stacked Bar** 또는 **Donut Chart**
- 색상:
  - 2xx: `bg-emerald-500`
  - 3xx: `bg-blue-500`
  - 4xx: `bg-amber-500`
  - 5xx: `bg-red-500`
- 범례: 각 코드별 카운트 표시

#### 4.2.11 Errors Table
- 컬럼: Error Type, Count, Percentage
- Error Type 색상 코딩:
  - Timeout: `text-amber-600`
  - CORS: `text-purple-600`
  - Network: `text-red-600`
  - HTTP 4xx: `text-amber-600`
  - HTTP 5xx: `text-red-600`

#### 4.2.12 Export Buttons
- 버튼 그룹: Copy Summary / JSON / CSV
- 아이콘: Copy, Download, FileJson, FileSpreadsheet
- 스타일: Secondary button (`border border-gray-300 dark:border-gray-600`)

## 7. 성능 고려사항

### 7.1 Web Worker 아키텍처

- **메인 스레드**: UI 렌더링, 사용자 입력 처리
- **Worker 스레드**: 부하 생성, 요청 스케줄링, 메트릭 집계
- **통신**: postMessage로 진행률/결과 스트리밍

### 7.2 메모리 관리

- per-request 데이터는 **전량 저장 금지**
- 기본: 집계 메트릭만 유지
- 옵션: per-request는 **샘플링** (1/N) 또는 최대 1,000개 제한
- Latency percentile 계산: 제한된 샘플 배열 + 정렬 방식

### 7.3 Rate Limiting 구현

- setTimeout 기반 토큰 버킷 (Worker 내부)
- 브라우저 타이머 정확도 한계 명시 (문구로 안내)

### 7.4 큰 응답 처리

- `collectResponseBytes=false` 시 스트림 빠르게 닫기 (`response.body?.cancel()`)
- 응답 바디 무시 옵션으로 메모리/네트워크 부담 최소화

## 8. 보안 및 프라이버시

### 8.1 데이터 흐름

- 모든 요청은 브라우저에서 직접 대상 서버로 전송
- tools.yowu.dev 서버는 요청/응답 데이터에 접근하지 않음
- 사용자 IP가 대상 서버에 직접 노출됨 (경고 문구 표시)

### 8.2 민감정보 처리

- localStorage에 인증정보/토큰 저장: **기본 OFF**
- 공유 링크에 민감정보 포함: **기본 OFF**, opt-in 필요
- Export JSON/CSV: 헤더/바디 포함 여부 토글 제공

### 8.3 CORS 에러 분류

- CORS 에러는 "네트워크 오류"와 명확히 분리
- CORS 차단 시 전용 안내 메시지 표시
- Extension 연동 안내 (있는 경우)

### 8.4 Companion Extension 연동 (선택)

- Extension 설치 시: `chrome.runtime.sendMessage`로 background fetch
- CORS 우회 + 쿠키 포함 요청 지원
- 웹앱은 Extension 없이도 동작 (CORS 가능한 대상만)

## 9. 엣지 케이스 처리

### 9.1 입력 검증

| 케이스 | 처리 |
|--------|------|
| `concurrency > n` | 에러: "Concurrency cannot exceed total requests" |
| `n = 0` 또는 `z = 0` | 에러: "Must specify at least 1 request or 1 second" |
| URL 없음 | Run 버튼 비활성화 |
| 잘못된 URL 형식 | 경고: "Invalid URL format" |
| HEAD/GET + Body | 경고 표시 + Body 무시 |

### 9.2 실행 중 처리

| 케이스 | 처리 |
|--------|------|
| Stop 클릭 | AbortController로 진행 중인 모든 요청 취소, Partial 결과 표시 |
| Duration 만료 | 자동 중지, 진행 중인 요청 완료 대기 (최대 timeout) |
| 탭 닫기/새로고침 | beforeunload 경고 (실행 중인 경우) |
| 네트워크 끊김 | 에러 카운트 증가, 테스트 계속 진행 |

### 9.3 브라우저 제약

| 케이스 | 처리 |
|--------|------|
| Redirect | 브라우저 기본 follow 동작, 비활성화 불가 명시 |
| HTTP/2 | 브라우저 자동 협상, 강제 선택 불가 명시 |
| Keep-alive | 브라우저 관리, 제어 불가 명시 |
